# numpy-practice

## Purpose

NumPy fluency, aimed specifically at the parts that matter once "array"
turns into "tensor": dtype/memory footprint, indexing, broadcasting,
vectorized speed, and treating an image as what it actually is, a 3D array.

## Files

| File | Description |
|---|---|
| `01_arrays_dtypes.py` | Array creation, dtype, shape. Measures real `nbytes` for float64 vs float32 vs int8 on 1M values, the seed of the quantization story. |
| `02_indexing_slicing.py` | Slicing, boolean masks, fancy indexing: cropping, thresholding, selecting rows. |
| `03_broadcasting.py` | Broadcasting rules with concrete shapes, including a per-channel image normalization example and a deliberate broadcasting failure. |
| `04_vectorization_bench.py` | Times a pure-Python loop vs a vectorized NumPy op on 2,000,000 elements and prints the measured speedup. |
| `05_image_as_array.py` | Loads the shared sample image, inspects shape/dtype, normalizes to `[0,1]`, explains HWC layout, adds a batch dimension. |

## Key concepts covered

- dtype determines memory footprint, and memory footprint is what changes when
  a model gets quantized.
- Slicing/masking/fancy indexing as the vocabulary for image cropping and
  filtering predictions by confidence.
- Broadcasting rules: same-shape isn't required, only trailing dimensions
  need to match (or be 1).
- Vectorization isn't a style preference, it's a measured performance
  difference.
- An image is `(Height, Width, Channels)`, uint8 `[0,255]` on disk, and models
  expect it normalized to float `[0,1]` with a leading batch dimension.

## How to run

```bash
python opencv-experiments/generate_sample_image.py   # generates sample.jpg, run once
python numpy-practice/01_arrays_dtypes.py
python numpy-practice/02_indexing_slicing.py
python numpy-practice/03_broadcasting.py
python numpy-practice/04_vectorization_bench.py
python numpy-practice/05_image_as_array.py
```

`05_image_as_array.py` depends on `opencv-experiments/sample.jpg` (a small
synthetic image committed to the repo, generated by
`opencv-experiments/generate_sample_image.py`), since Phase 2 comes before the
OpenCV phase but needs a real image to inspect.

## Results

From `01_arrays_dtypes.py`, storing 1,000,000 weights:

| dtype | nbytes | size |
|---|---|---|
| float64 | 8,000,000 | 7.63 MB |
| float32 | 4,000,000 | 3.81 MB |
| int8 | 1,000,000 | 0.95 MB |

float64 to float32 shrinks storage 2.0x. float32 to int8 shrinks storage
4.0x (8x total from float64). This is the same shrink ratio, per weight
across millions of weights, behind the model file sizes in
`tensorflow-lite/`.

From `04_vectorization_bench.py`, summing `2,000,000` elementwise-transformed
values:

| Method | Time |
|---|---|
| Pure Python loop | 0.1359 s |
| NumPy vectorized | 0.0187 s |

**Measured speedup: 7.3x.** This varies run to run (repeated runs on this
machine ranged roughly 7x-10x) since it's printed fresh each time the script
runs, not hardcoded. The point is the order of magnitude, not the exact
digit.

## Why this matters for Edge AI

A model's weights are a NumPy array (or a tensor with identical memory
semantics), and dtype directly determines RAM footprint, load time, and how
much data has to move through the CPU/NPU per inference.
`01_arrays_dtypes.py`'s measured 4x shrink from float32 to int8 isn't a toy
exercise: it's the exact mechanism `tensorflow-lite/02_convert_quantized.py`
uses to make a model small enough, and fast enough, to run well on a
Raspberry Pi. The vectorization speedup in `04_vectorization_bench.py`
matters even more on a Pi's weaker CPU than on a development laptop, since
an unvectorized preprocessing loop can dominate total inference latency.

## Common mistakes / gotchas

- `np.zeros()` and `np.random.default_rng().standard_normal()` default to
  float64, not float32. If you don't explicitly cast, your "float32 model"
  may quietly be doing float64 math somewhere in preprocessing.
- Boolean mask indexing (`arr[mask]`) returns a new array; it does not modify
  in place unless you assign into the mask, e.g. `arr[mask] = 0`.
- Broadcasting silently "works" as long as trailing dimensions match or are
  1, which is convenient but can also silently broadcast against the wrong
  axis if you're not careful about shape order (see the channel_mean example
  in `03_broadcasting.py`, which relies on the mean being shape `(3,)` to line
  up with the image's last axis).
- Casting `float64 * 127` straight to `int8` (as in `01_arrays_dtypes.py`)
  can overflow or wrap silently for values outside int8's range. Real
  quantization code needs an explicit scale/zero-point calibration step,
  which is exactly what `tensorflow-lite/02_convert_quantized.py` does
  properly with a representative dataset.
